# 项目代码审查规范与评分标准 (v2.0)

## 1. 问题严重等级定义

- 🔴 严重（Severe）
  - 定义：系统崩溃、数据丢失、安全漏洞、核心业务逻辑错误。
  - 示例：SQL 注入、鉴权缺失、死循环、关键金额计算错误。
  - 处理：必须修复，阻断合并（Block Merge）。

- 🟡 中等（Moderate）
  - 定义：性能隐患、可维护性差、非核心逻辑错误、违反架构原则。
  - 示例：N+1 查询、大量重复代码、硬编码配置、缺乏必要异常处理。
  - 处理：原则上必须修复；特殊情况需 Tech Lead 批准。

- 🟢 轻微（Minor）
  - 定义：风格、命名、注释、拼写、非关键最佳实践建议。
  - 示例：命名不统一、格式问题、注释错字、可简化写法。
  - 处理：建议修复，可协商后续迭代处理。

## 2. 代码质量评分系统

### 2.1 五星评级

- ⭐⭐⭐⭐⭐（90-100）：完美（Excellent）-> Approve
- ⭐⭐⭐⭐（80-89）：良好（Good）-> Approve with suggestions
- ⭐⭐⭐（60-79）：合格（Acceptable）-> Request Changes
- ⭐⭐（40-59）：较差（Poor）-> Request Changes
- ⭐（0-39）：极差（Disaster）-> Reject/Close

### 2.2 扣分细则（推荐）

- 初始分：100
- 严重问题：每个 -20
- 中等问题：每个 -8（可在 5-10 范围按影响调整）
- 轻微问题：每个 -2（可在 1-2 范围按影响调整）
- 最低分：0

## 4. 检查清单

### 🔴 必查（严重）

- [ ] 安全性：是否存在 SQL 注入、XSS、越权访问？
- [ ] 数据完整性：是否存在数据丢失风险？
- [ ] 核心逻辑：关键分支与边界值处理是否正确？
- [ ] 稳定性：是否存在明显死循环/泄漏/崩溃风险？

### 🟡 重点（中等）

- [ ] 性能：是否存在明显性能反模式？
- [ ] 可维护性：是否函数过长、重复代码多？
- [ ] 异常处理：是否吞异常或日志信息不足？
- [ ] 测试覆盖：核心路径是否有测试保障？

### 🟢 建议（轻微）

- [ ] 规范性：命名与风格是否一致？
- [ ] 文档：注释是否清晰准确？
- [ ] 整洁度：是否有无用 import/注释代码？

## 5. 争议解决机制

1. 先沟通并说明评级依据。
2. 无法达成一致时，由架构师/Tech Lead 仲裁。
3. 对暂不修复的中等问题，必须登记技术债（Jira/Trello）并标注 TODO。
