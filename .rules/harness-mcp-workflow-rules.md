# Resume Builder Harness 与 MCP 工作流规则

## 1. 规则定位

本文档用于定义 `resume-builder` 仓库在日常开发、联调、审查、排障与知识回写过程中，AI 必须如何使用 Harness 与 MCP 能力。

本文档属于仓库级强制规则，适用于以下场景：

1. 前端开发与交互验证。
2. `python-ai-backend/` 开发与 OpenAI 能力接入。
3. `spring-ai-backend/` 开发与 OpenAI 能力接入。
4. PR / Issue / Review / Checks 协作。
5. 文档沉淀、规则回写与长期偏好管理。

如无更高优先级用户指令，不得绕过本文档中定义的 MCP 使用规则。

## 2. 当前 MCP 基线

当前已接入或默认可用的 MCP 能力，按仓库开发价值排序如下：

1. `openaiDeveloperDocs`
   - 用于查询 OpenAI 官方最新文档。
2. `playwright`
   - 用于浏览器级页面与交互验证。
3. `github`
   - 用于读取 GitHub Issue、PR、Review、Checks 与远程协作上下文。
4. `memory`
   - 用于记录用户长期偏好或稳定的个人协作习惯。
5. `pencil`
   - 用于 `.pen` 设计稿、设计系统或视觉稿协作。

## 3. MCP 总原则

### 3.1 先判断任务阶段，再决定 MCP

AI 在开发任务中，不得“看到 MCP 就用”，必须先判断当前处于哪个阶段：

1. 任务受理与背景确认。
2. 外部文档调研。
3. 设计稿或视觉资产确认。
4. 实现与联调。
5. 页面 / 交互验证。
6. PR / Review / Checks 协作。
7. 知识回写与长期偏好记录。

只有当前阶段与 MCP 能力匹配时，才能调用对应 MCP。

### 3.2 Repo 是系统记录，memory 不是

长期有效的项目事实、规则、流程、接口契约、目录职责，必须写回仓库，而不是只写进 `memory`。

必须写回仓库的内容包括：

1. 新的规则与协作流程。
2. 新的目录边界与模块职责。
3. 新的启动方式与排障方式。
4. 前后端共享契约变化。
5. OpenAI 接入策略与稳定配置说明。

`memory` 只允许记录以下内容：

1. 用户长期个人偏好。
2. 用户常用的工作习惯。
3. 明确不会替代仓库文档的辅助记忆。

### 3.3 MCP 不替代仓库规则

无论 MCP 返回什么信息，AI 都必须继续遵守：

1. `AGENTS.md`
2. `.rules/` 下对应规则文档
3. 根 `README.md`
4. 子模块 `README.md`
5. 当前代码目录边界

### 3.4 新功能必须先拆分再执行

凡是涉及以下类型的工作，默认都视为“新功能或行为变更任务”：

1. 新功能开发。
2. 影响页面、接口、业务行为的优化。
3. 影响交互、数据流或输出结果的重构。
4. 需要多步协作才能完成的文档化流程改造。

对这类任务，AI 必须先做任务拆分，再进入实现。

强制要求如下：

1. 必须先把目标拆成一条条细化任务，而不是直接开始改代码。
2. 每条细化任务必须足够具体，能够独立判断是否完成。
3. 细化任务必须有顺序，前置任务未完成时不得跳过直接做后置任务。
4. 用户描述不清晰时，AI 必须先给出一版“初步任务 todo list”用于澄清需求，而不是自作主张补全模糊需求。
5. 用户确认或修正 todo list 后，AI 才能进入正式执行阶段。

### 3.5 验收必须先定义，再执行，再闭环

新功能或行为变更任务，必须先定义验收细节，再开始顺序执行。

强制要求如下：

1. 在用户确认细化任务后，AI 必须基于这些细化任务生成对应的验收细节 list。
2. 验收细节必须可检查、可判断，不能写成空泛目标。
3. AI 完成全部细化任务后，必须按验收细节逐条验证。
4. 每条验收项都必须有明确状态，至少包含：`待验证`、`通过`、`不通过`。
5. 只要存在任意一条验收项为 `不通过`，就不得声称任务完成。
6. 出现 `不通过` 时，AI 必须重新检查相关逻辑、修正问题，并重新进行全量验收，直到所有验收项都变为 `通过`。

## 4. 各 MCP 的强制使用规则

### 4.1 `openaiDeveloperDocs`

以下场景必须优先使用 `openaiDeveloperDocs`，不得直接凭记忆回答或改代码：

1. 涉及 OpenAI `Responses API`、`Agents SDK`、`Realtime`、`MCP`、模型选择、参数差异、接口迁移。
2. 涉及 `python-ai-backend/` 或 `spring-ai-backend/` 中 OpenAI 上游能力接入或调整。
3. 涉及“最新 / 当前 / 推荐 / 官方”这类 OpenAI 能力判断。

典型适用场景：

1. 调整聊天、流式输出、语音转写、Realtime 临时密钥相关实现。
2. 判断该使用哪种 OpenAI API 或 SDK。
3. 修改 OpenAI 请求参数、模型名、路径或能力组合。

禁止行为：

1. 在 OpenAI 能力存在时效性的前提下只凭记忆改代码。
2. 把非官方二手资料当作 OpenAI 接入依据。

### 4.2 `playwright`

以下场景必须优先使用 `playwright` 做浏览器级验证，不能只靠静态读代码判断完成：

1. 前端页面结构、样式、交互、按钮状态、弹窗、表单、流式渲染发生变化。
2. AI 优化面板、AI 面试页、模板选择页、简历预览区发生变化。
3. 用户明确要求检查页面是否正常、是否可点击、是否有视觉问题。

典型适用场景：

1. `src/components/`
2. `src/templates/`
3. `src/App.vue`
4. `src/services/` 中影响页面交互的调用链

禁止行为：

1. 对明显的前端交互改动不做浏览器验证就声称完成。
2. 仅以 `npm run lint` 通过作为页面正确的唯一依据。

补充约束：

1. `playwright` 用于页面与交互验证，不替代后端业务正确性判断。
2. 如果环境不具备运行条件，必须明确说明未完成浏览器验证，并给出原因。

### 4.3 `github`

以下场景必须优先使用 `github` 读取远程协作上下文：

1. 用户提到 Issue、PR、Review、Checks、评论、远程分支、合并状态。
2. 需要理解代码评审意见来源，而不是只看本地工作区。
3. 需要判断远端 PR 状态、CI 结果、Reviewer 反馈。

典型适用场景：

1. 修复 PR review 问题。
2. 查看远程 issue 中的需求背景。
3. 查看某个 PR 的改动面和 check 状态。

禁止行为：

1. 在需要远程上下文时，只凭本地 git 记录推断 PR/Issue 结论。
2. 将 `github` MCP 返回的信息视为可绕过仓库规则的依据。

### 4.4 `memory`

以下场景才允许使用 `memory`：

1. 用户重复表达且希望长期保留的个人偏好。
2. 用户的开发环境偏好，如常用 `cmd`、常用命令顺序。
3. 明确不会替代项目事实的协作偏好。

典型适用场景：

1. 记住用户更偏好 `cmd` 而不是 PowerShell。
2. 记住用户更关注 PR 审查结果与交互闭环。

禁止行为：

1. 用 `memory` 存放项目架构事实、接口契约、数据库约束、规则文档内容。
2. 把应该写入仓库的长期知识只存在 `memory` 里。

### 4.5 `pencil`

以下场景可以使用 `pencil`，但不是默认必选：

1. 仓库中存在 `.pen` 文件或设计稿工作流。
2. 用户要求读取、修改或生成设计稿。
3. 需要比纯文本更精确地管理页面结构或设计系统。

典型适用场景：

1. 简历模板视觉方案设计。
2. AI 面试页视觉布局调整。
3. 设计系统组件或页面草图的结构化管理。

禁止行为：

1. 当前任务没有设计稿需求时，强行引入 `pencil` 增加复杂度。
2. 用设计稿工具替代前端真实页面验证。

## 5. 按开发流程的 MCP 使用顺序

### 5.1 任务受理阶段

默认顺序：

1. 读取 `AGENTS.md`
2. 读取 `.rules/` 对应规则
3. 读取根 `README.md`
4. 读取相关子模块 `README.md`
5. 判断是否需要 `github`
6. 判断是否需要 `openaiDeveloperDocs`

此阶段强制要求：

1. 先做仓库上下文判断，再决定是否调用外部 MCP。
2. 如果是远程协作任务，优先补 `github` 上下文。
3. 如果是 OpenAI 相关任务，优先补 `openaiDeveloperDocs` 上下文。

### 5.2 方案设计与调研阶段

适用要求：

1. OpenAI 接入与模型能力判断，必须先查 `openaiDeveloperDocs`。
2. 设计稿任务，按需使用 `pencil`。
3. 远程需求背景缺失时，使用 `github`。

### 5.3 新功能需求澄清与初步 todo list 阶段

当任务属于新功能或行为变更时，必须先经过这一阶段。

强制要求：

1. 用户需求清晰度不足时，AI 必须先输出一版“初步任务 todo list”。
2. 这版 todo list 的目标是帮助用户澄清范围、边界、优先级与非目标，而不是直接当作最终执行清单。
3. 在用户未确认前，AI 不得直接进入完整实现。
4. 用户确认后，AI 必须整理为正式的细化任务列表。

### 5.4 验收细节 list 生成阶段

在用户确认细化任务列表后，AI 必须生成验收细节 list。

强制要求：

1. 每条细化任务都应映射到一条或多条验收细节。
2. 验收细节必须具体到可以检查结果，而不是只写“功能正常”这类空泛描述。
3. 如果验收细节无法定义清楚，AI 必须先回到需求澄清，而不是直接开始实现。

### 5.5 需求文档生成阶段

在正式执行前，AI 必须生成需求文档 `.md` 文件，作为该任务的系统记录。

默认要求如下：

1. 默认路径为 `docs/requirements/YYYY-MM-DD-<topic>.md`。
2. 若用户明确指定其他位置，以用户指令为准。
3. 需求文档至少必须包含以下内容：
   - 任务背景
   - 目标
   - 范围
   - 非目标
   - 细化任务清单
   - 验收细节 list
   - 任务执行状态
   - 验收状态
4. 细化任务清单必须使用可更新状态的格式。
5. 验收细节 list 必须为每一条预留状态字段。

推荐状态如下：

1. 细化任务状态：`待办`、`进行中`、`已完成`、`阻塞`
2. 验收状态：`待验证`、`通过`、`不通过`

### 5.6 实现阶段

实现阶段通常不强制调用 MCP，但必须保留前一阶段得到的约束结论。

此阶段要求：

1. 不得在实现中偏离通过 `openaiDeveloperDocs` 确认过的官方约束。
2. 不得在实现中偏离通过 `github` 获取到的需求 / review 结论。
3. 若实现中发现规则缺口，应记录为后续知识回写项。
4. AI 必须按需求文档中的细化任务顺序逐条执行，不得跳步声称整体完成。
5. 每完成一条细化任务，必须立即在需求文档中将该任务标记为 `已完成`，并同步记录必要的实现说明或验证说明。
6. 如执行中出现阻塞、范围变化或用户补充需求，必须先更新需求文档，再继续后续任务。

### 5.7 验收与测试阶段

此阶段是 MCP 使用最强制的阶段，也是任务闭环的最终判断阶段。

必须规则：

1. 前端可视变更优先用 `playwright`。
2. OpenAI 接入调整优先做官方文档一致性确认与本地允许范围内验证。
3. PR / Review 修复任务优先看 `github` 上的 review / checks 是否匹配。
4. 全部细化任务完成后，AI 必须对需求文档中的每一条验收细节逐条测试并更新状态。
5. 每一条验收细节都必须明确标记为 `通过` 或 `不通过`，不得留空声称整体通过。
6. 如果存在任意一条 `不通过`，AI 必须重新检查相关逻辑，修正问题后再次进行全量验收，而不是只口头说明“理论上已修复”。
7. 只有当验收细节 list 全部为 `通过` 时，AI 才能声称任务完成。

### 5.8 知识回写阶段

回写阶段默认不依赖 MCP，但应利用前面阶段得到的结果，把稳定知识落入仓库。

必须规则：

1. 规则写回 `.rules/`
2. 协作流程写回 `docs/`
3. 启动与联调说明写回 `README.md` 或子模块 `README.md`
4. 用户个人偏好如确有长期价值，再考虑写入 `memory`

### 5.9 审查与合并阶段

如果任务需要进入提交或审查流程，除遵守现有审查规则外，还必须满足：

1. 需求文档中的细化任务已经全部标记为 `已完成`。
2. 需求文档中的验收细节 list 已全部标记为 `通过`。
3. 若文档中仍存在 `待办`、`阻塞`、`待验证` 或 `不通过` 项，不得进入“任务已完成”的结论。

## 6. 需求文档模板要求

新功能或行为变更任务的需求文档，至少应符合以下结构：

```md
# <任务标题>

## 1. 背景

## 2. 目标

## 3. 范围

## 4. 非目标

## 5. 细化任务清单
- [ ] 任务 1
- [ ] 任务 2

## 6. 验收细节 list
- [待验证] 验收项 1
- [待验证] 验收项 2

## 7. 执行记录
- 2026-04-19：完成任务 1

## 8. 验收结果
- [通过] 验收项 1
- [不通过] 验收项 2
```

要求如下：

1. 细化任务清单必须是可更新的执行清单。
2. 验收细节 list 必须是可更新的验收清单。
3. 执行记录必须能反映每条细化任务的完成过程。
4. 验收结果必须反映最终逐条测试的结论。

## 7. 针对本仓库的标准场景

### 7.1 前端页面 / 模板开发

标准顺序：

1. 读取 `AGENTS.md` 与前端规则
2. 输出并确认细化任务清单
3. 生成需求文档与验收细节 list
4. 读取相关组件 / 模板 / 服务代码
5. 按顺序实现改动并更新文档状态
6. 执行允许的命令验证
7. 使用 `playwright` 验证页面与交互
8. 按验收细节 list 全量验收
9. 如有稳定变化，回写文档

### 7.2 Python AI 后端开发

标准顺序：

1. 读取 `AGENTS.md`、后端规则、Python AI 后端规则
2. 输出并确认细化任务清单
3. 生成需求文档与验收细节 list
4. 涉及 OpenAI 能力时先查 `openaiDeveloperDocs`
5. 按顺序实现代码改动并更新文档状态
6. 做允许范围内的健康检查或接口验证
7. 按验收细节 list 全量验收
8. 如接口或接入策略稳定变化，回写文档

### 7.3 Spring AI 后端开发

标准顺序：

1. 读取 `AGENTS.md`、后端规则、Spring AI 后端规则
2. 输出并确认细化任务清单
3. 生成需求文档与验收细节 list
4. 涉及 OpenAI 能力时先查 `openaiDeveloperDocs`
5. 涉及 SQL 时先确认 `mapper.xml` 与 SQL 目录规则
6. 按顺序实现代码改动并更新文档状态
7. 做允许范围内的健康检查或接口验证
8. 按验收细节 list 全量验收
9. 如接入策略稳定变化，回写文档

### 7.4 PR 审查与修复

标准顺序：

1. 先查 `github` 获取 PR、Review、Checks 上下文
2. 输出并确认细化修复任务清单
3. 生成需求文档与验收细节 list
4. 再读取本地代码与仓库规则
5. 按顺序实施修复并更新文档状态
6. 做允许范围内验证
7. 按验收细节 list 全量验收
8. 如规则不足以约束同类问题，补规则或 runbook

## 8. MCP 不可用时的兜底规则

如果某个 MCP 当前不可用，AI 必须遵守以下规则：

1. 明确说明哪个 MCP 不可用。
2. 明确说明原本应在哪个流程使用它。
3. 明确说明采用了什么兜底方案。
4. 不得假装已经完成对应 MCP 验证。

允许的兜底方式：

1. `openaiDeveloperDocs` 不可用时，才允许使用 OpenAI 官方站点作为兜底。
2. `playwright` 不可用时，可退回人工说明未做浏览器验证。
3. `github` 不可用时，只能基于本地已存在信息继续，且必须说明远程上下文缺失。
4. `memory` 不可用时，直接回写仓库文档，不影响项目事实记录。

## 9. 规则优先级

本文档在以下范围内具有强制优先级：

1. 开发流程中的 MCP 选择与使用顺序。
2. OpenAI 文档调研的必用条件。
3. 前端交互验证的必用条件。
4. 远程协作上下文获取的必用条件。
5. `memory` 的边界限制。

如与其他规则冲突，按以下顺序处理：

1. 用户明确指令
2. `AGENTS.md`
3. 本文档
4. 其他一般性协作文档

## 10. 结论

本仓库中的 MCP 不应被当成“可有可无的外挂工具”，而应被纳入标准开发流程：

1. `openaiDeveloperDocs` 负责官方事实。
2. `playwright` 负责页面事实。
3. `github` 负责协作事实。
4. `memory` 负责个人偏好。
5. `pencil` 负责设计稿事实。

只有把它们放在正确的流程节点上使用，Harness 才会真正稳定，而不是停留在“装了几个工具”的阶段。
