# Resume Builder 前端强制规则

## 1. 规则定位

本文档用于定义 `resume-builder` 仓库前端范围内的产品背景、目标边界、目录协作方式、页面设计要求、代码分层要求与交付限制。

本仓库前端不是普通的简历模板展示站，而是一个服务求职场景的交互前台，承担 **简历编辑、简历优化、AI 面试** 的核心用户体验，目标是帮助求职者更快、更精准地完成简历准备与面试通关。

凡涉及前端页面、新增组件、模板扩展、交互调整、状态管理、接口调用定义、服务编排和样式设计的改动，默认都必须遵守本文档。

注意：
- 本文档位于 `.rules/` 下，属于仓库级强制规则，不视为普通说明文档。
- 本文档适用于前端相关目录与文件，包括但不限于 `src/` 下的页面、组件、模板、接口定义、服务与状态管理代码。
- 如当前实现与本文档冲突，后续改动应优先向本文档靠拢；如确需例外，应先更新规则文档，再改代码。

---

## 2. 产品背景与前端目标

前端必须始终围绕以下业务目标展开：

1. 简历编辑
   - 为用户提供结构化、可视化、可导出的简历编辑体验。
   - 保证模板、版式、字段组织和预览行为稳定一致。

2. 简历优化
   - 提供 AI 优化交互入口，帮助用户对简历内容进行润色、重写、提炼与表达优化。
   - 输出体验必须服务真实求职与面试场景，禁止页面表达偏离业务主线。

3. AI 面试
   - 提供 AI 面试入口、过程交互、会话展示与反馈承接界面。
   - 界面重点应是信息清晰、节奏稳定、反馈可理解。

4. 一体化体验
   - 简历编辑、简历优化、AI 面试必须在交互语言、视觉风格和信息架构上保持统一产品感。
   - 禁止生成与当前产品无关、无业务承接、仅为展示效果存在的页面。

---

## 3. 前端执行原则

### 3.1 规则读取要求

1. 处理前端任务时，必须先阅读并遵守 `.rules/` 下与前端相关的规则文档。
2. `.rules/` 下规则文档与 `AGENTS.md` 共同构成仓库级强制约束；如无更高优先级用户指令，不得绕过。
3. 前端改动除遵守本文档外，还必须同步遵守 `.rules/global-rules.md` 与 `.rules/code-conventions.md`。

### 3.2 目标导向原则

1. 所有页面、组件、交互和模板都应围绕求职、简历、面试主线展开。
2. 禁止脱离现有业务语义，随意生成无关模块、无关页面或演示性代码。
3. 新增功能必须与现有目录职责、主题风格、业务流程保持一致。

### 3.3 文件作者标识约束

1. 新增或修改的前端文件必须标记作者信息，例如注释头或元数据。
2. 作者必须为 `jf`。
3. 禁止出现作者为 `ai` 的标识，包括但不限于 `author: ai`、`作者：ai`、`created by ai`。

---

## 4. 前端目录与分层约束

当前前端目录必须围绕现有结构演进，包括但不限于：

```text
src/
├─ api/
├─ assets/
├─ components/
├─ config/
├─ constants/
├─ services/
├─ stores/
└─ templates/
```

强制要求：
- 编写前端代码时，必须严格遵守现有目录分级规则，优先复用现有目录与子目录语义。
- 禁止绕开当前结构，随意在外层新增新的平级目录。
- 禁止在非该功能目录下新增不属于该功能的文件。
- 如确需扩展目录，优先在现有目录内按功能继续细分；无法归类时，必须先更新规则文档。

---

## 5. 页面设计与视觉一致性约束

当 AI 设计新页面、新模块界面或大幅调整页面样式时，必须遵守以下要求：

1. 必须调用 MCP `pencli` 相关设计能力产出页面样式方案。
2. 必须使用设计类技能：
   - `frontend-design`
   - `ui-ux-pro-max`
3. 若当前环境缺失上述技能中的任意一项，必须先明确说明，再采用最接近的可用设计流程继续执行，禁止静默跳过。
4. 设计结果必须遵守当前页面已经建立的主题风格、颜色配比和样式规范。
5. 禁止随意生成与当前产品不同风格的新页面。
6. 禁止为了“更酷”或“更有设计感”而破坏现有模板体系、品牌感和交互一致性。

---

## 6. 前端代码组织约束

1. AI 每次生成的单个前端代码文件内容不得超过 `1000` 行。
2. 超过 `1000` 行时，必须按功能、类型或职责进行拆分，禁止继续堆叠在单文件中。
3. 组件、页面、状态、服务、接口定义应尽量保持单一职责。
4. 禁止把页面结构、业务逻辑、接口定义、数据转换全部塞进同一个文件。

---

## 7. 前端 API 与 Service 边界

1. 前端 `api/` 层只负责接口请求定义，不承载业务逻辑。
2. 与接口调用相关的业务编排、数据加工、组合调用、状态联动逻辑，必须放在 `services/` 或对应职责层中。
3. 禁止在 `api/` 文件中编写复杂判断、流程控制、状态处理、兜底策略。

---

## 8. 前端验证与交付要求

1. AI 每次完成前端代码编写后，必须严格执行 `npm run lint`。
2. `lint` 必须通过，才可认为该次前端改动完成。
3. 如改动影响类型或构建稳定性，可继续使用现有命令进行补充验证，但不得新增测试代码。

允许使用以下方式验证：
- `npm run type-check`
- `npm run lint`
- `npm run build`
- `npm run build-only`
- 本地手工验证，例如 UI 操作、接口调用、日志观察
- 一次性命令行验证，不落盘到项目文件，不生成新代码文件

---

## 9. 测试代码约束

1. 禁止在项目中新增任何测试代码。
2. 禁止修改现有测试代码。
3. 禁止为测试目的新增以下内容：
   - `src/test`、`__tests__`、`tests` 目录下任何文件
   - `*.test.*`、`*.spec.*` 文件
   - 测试专用脚本、测试夹具（fixture）、测试桩（mock）文件
4. 禁止为了验证问题而提交“临时测试代码”。

---

## 10. 结论

前端的一切改动，都必须服务“帮助求职者更快、更精准地进行面试并提升通关效率”这一核心目标。

因此必须同时保证：

- 产品方向不跑偏。
- 页面风格不失控。
- 目录分层不混乱。
- 模板体系不被破坏。
- 接口定义与业务逻辑边界清晰。
- AI 产出的前端代码可持续维护。
